# Foodiya API

## Architecture

```
Foodiya.API  →  Foodiya.Application  →  Foodiya.Domain
                   Foodiya.Infrastructure →  Foodiya.Domain
```

| Layer              | Responsibility                                                                          |
| ------------------ | --------------------------------------------------------------------------------------- |
| **API**            | Controllers, middleware, startup / DI composition root                                  |
| **Application**    | Use-case services, factories, orchestration                                             |
| **Domain**         | Domain models (incl. scaffolded entities), DTOs, interfaces, specifications, exceptions |
| **Infrastructure** | EF Core DbContext, repository implementations, data access                              |

## Local Configuration

Do **not** commit real connection strings.

Use one of these local-only options:

1. Create `Foodiya.API/appsettings.Local.json`
2. Set the environment variable `ConnectionStrings__DefaultConnection`

Example `appsettings.Local.json`:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=<server>;Initial Catalog=<database>;User ID=<user>;Password=<password>;Encrypt=True;TrustServerCertificate=False;Connection Timeout=30;"
  }
}
```

## EF Core Scaffolding

Scaffolding generates two things into two separate projects:

| Artifact                 | Project & Path                         | Namespace                     |
| ------------------------ | -------------------------------------- | ----------------------------- |
| `AutoGeneratedDbContext` | `Foodiya.Infrastructure/Data/`         | `Foodiya.Infrastructure.Data` |
| Entity POCOs             | `Foodiya.Domain/Models/AutoGenerated/` | `Foodiya.Domain.Models`       |

Run from the solution root:

```bash
dotnet ef dbcontext scaffold \
  "<connection-string>" \
  Microsoft.EntityFrameworkCore.SqlServer \
  -p Foodiya.Infrastructure \
  -s Foodiya.API \
  -o "../Foodiya.Domain/Models/AutoGenerated" \
  --context-dir "Data" \
  -c AutoGeneratedDbContext \
  -n Foodiya.Domain.Models \
  --context-namespace Foodiya.Infrastructure.Data \
  --no-onconfiguring \
  -f
```

### Flags explained

| Flag                                              | Purpose                                                         |
| ------------------------------------------------- | --------------------------------------------------------------- |
| `-p Foodiya.Infrastructure`                       | The project that owns the DbContext (EF tools run against this) |
| `-s Foodiya.API`                                  | Startup project (provides the connection string at design-time) |
| `-o "../Foodiya.Domain/Models/AutoGenerated"`     | Entity output path — relative to `-p`, lands in **Domain**      |
| `--context-dir "Data"`                            | DbContext output folder inside Infrastructure                   |
| `-c AutoGeneratedDbContext`                       | DbContext class name                                            |
| `-n Foodiya.Domain.Models`                        | Namespace for generated entity classes                          |
| `--context-namespace Foodiya.Infrastructure.Data` | Namespace for the DbContext                                     |
| `--no-onconfiguring`                              | Omits `OnConfiguring` — connection string is injected via DI    |
| `-f`                                              | Force overwrite existing files                                  |

### After scaffolding

1. **Build** — `dotnet build Foodiya.API.sln` to verify no breaks.
2. **Review** — Check that `AutoGeneratedDbContext.cs` still has `using Foodiya.Domain.Models;` at the top (EF adds it automatically since entities live in that namespace).
3. **No manual moves needed** — entities land in Domain, DbContext lands in Infrastructure, matching the existing project layout.
4. **Domain stays EF-free** — Without `--data-annotations`, scaffolded entities are pure POCOs (only `System.ComponentModel.DataAnnotations` attributes). All index, unicode, and relationship config lives in Fluent API inside `AutoGeneratedDbContext.cs`. If EF adds `using Microsoft.EntityFrameworkCore;` or EF-specific attributes (`[Index]`, `[Unicode]`, `[PrimaryKey]`, `[InverseProperty]`) to the generated models, remove them and move the config to Fluent API.

## Azure Functions (`Foodiya.Functions`)

Serverless functions that run alongside the main API for background/scheduled/standalone tasks.

### Functions Overview

| Function                      | Trigger                                      | Description                                                                                                      |
| ----------------------------- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `DailyRecipeStatsFunction`    | ⏰ Timer (`0 0 2 * * *` — daily at 2 AM UTC) | Computes daily analytics: total recipes, likes today, top-5 liked, recent recipes, stats by cuisine & difficulty |
| `WelcomeEmailFunction`        | 📧 Queue (`welcome-emails`)                  | Processes welcome email messages enqueued by the API after user registration                                     |
| `NutritionCalculatorFunction` | 📊 HTTP (`POST /api/nutrition/calculate`)    | Calculates total & per-serving calories from a list of ingredients                                               |

### Prerequisites

```bash
# Install Azure Functions Core Tools (via npm)
npm install -g azure-functions-core-tools@4

# Install Azurite (local Azure Storage emulator — needed for Timer & Queue triggers)
npm install -g azurite
```

### Configuration

Edit `Foodiya.Functions/local.settings.json` with your connection string:

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "SqlConnectionString": "<your-sql-connection-string>",
    "WelcomeEmailFromAddress": "noreply@foodiya.app",
    "WelcomeEmailFromName": "Foodiya"
  }
}
```

> ⚠️ Do **not** commit `local.settings.json` with real credentials.

### Run Locally

```bash
# 1. Start Azurite (in a separate terminal — keep it running)
azurite --silent --location /tmp/azurite --debug /tmp/azurite/debug.log

# 2. Build the Functions project
cd Foodiya.Functions
dotnet build -o bin/output

# 3. Start the Functions host
func start --prefix bin/output
```

You should see output like:

```
Functions:

        NutritionCalculate: [POST] http://localhost:7071/api/nutrition/calculate
        DailyRecipeStatsFunction: timerTrigger
        WelcomeEmailFunction: queueTrigger
```

### Test the HTTP Function

In a **separate terminal**:

```bash
curl -s -X POST http://localhost:7071/api/nutrition/calculate \
  -H "Content-Type: application/json" \
  -d '{
    "servings": 4,
    "ingredients": [
      { "ingredientId": 1, "quantityGrams": 200 },

  }'
```

### Enqueue a Welcome Email (from the API)

After user registration in your API, enqueue a message to the `welcome-emails` queue:

```csharp
var queueClient = new QueueClient(connectionString, "welcome-emails");
await queueClient.CreateIfNotExistsAsync();

var message = new WelcomeEmailMessage
{
    UserId        = newUser.Id,
    Email         = newUser.Email,
    FirstName     = newUser.FirstName,
    UserName      = newUser.UserName,
    RegisteredAtUtc = DateTime.UtcNow
};

var json = JsonSerializer.Serialize(message);
var base64 = Convert.ToBase64String(Encoding.UTF8.GetBytes(json));
await queueClient.SendMessageAsync(base64);
```
